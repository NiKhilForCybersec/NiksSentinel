SPL Fundamentals

Introduction to Search Processing Language - the power behind Splunk searches

üìò Chapter Overview

SPL (Search Processing Language) is Splunk's query language. Mastering SPL is essential for effective security analysis. This chapter covers the fundamentals: search syntax, time modifiers, boolean operators, field searching, and the basics of search optimization.

Anatomy of a Splunk Search

Every Splunk search follows a pipeline structure: 

# Basic search structure
search_terms | command1 | command2 | command3

# Example
index=security sourcetype=WinEventLog:Security EventCode=4625 
| stats count by user 
| sort -count 
| head 10

Search Pipeline Concept

Data flows through the pipeline:

[Raw Events] ‚Üí [Filter] ‚Üí [Transform] ‚Üí [Visualize]
     ‚îÇ            ‚îÇ            ‚îÇ            ‚îÇ
   index=     where/search   stats        table
   source                    chart        head
   sourcetype               timechart     sort

Search Components

# Search components breakdown:

index=security           # WHERE to search (most important!)
sourcetype=WinEventLog   # What type of data
EventCode=4625           # Field=value filter
"failed login"           # Keyword search in _raw
| stats count by user    # Transformation command
| where count > 10       # Filter results
| sort -count            # Order results
| head 5                 # Limit results

Time Range Modifiers

Time is critical for efficient searches. Always specify time ranges:

Relative Time Modifiers

# Relative time syntax: earliest/latest
earliest=-24h          # Last 24 hours
earliest=-7d           # Last 7 days
earliest=-1mon         # Last 1 month
earliest=-1y           # Last 1 year

# Combined modifiers
earliest=-24h latest=now      # Last 24 hours
earliest=-7d@d latest=@d      # Last 7 days, snapped to day
earliest=-1h@h latest=now     # Last hour, snapped to hour

# Snap-to modifiers (@)
earliest=-1d@d         # Yesterday at midnight
earliest=@d            # Today at midnight
earliest=-1w@w         # Start of last week (Sunday)
earliest=-1mon@mon     # Start of last month

Time Unit Reference

Unit Meaning Example
sseconds-30s
mminutes-15m
hhours-24h
ddays-7d
wweeks-1w
monmonths-3mon
yyears-1y

Absolute Time

# Absolute time format: MM/DD/YYYY:HH:MM:SS
earliest="01/15/2024:00:00:00" latest="01/15/2024:23:59:59"

# ISO format
earliest="2024-01-15T00:00:00" latest="2024-01-15T23:59:59"

# Epoch time
earliest=1705276800 latest=1705363199
‚úì Best Practice

Always specify time ranges. Without time limits, Splunk searches all data, which is slow and expensive. For security investigations, start with narrow time ranges and expand if needed. Use the time picker in the UI or inline time modifiers in your search.

Boolean Operators

AND, OR, NOT

# AND - implicit (space between terms)
index=security user=admin action=failed
# Equivalent to:
index=security user=admin AND action=failed

# AND - explicit
index=security user=admin AND src_ip=10.0.0.50

# OR
index=security (EventCode=4624 OR EventCode=4625)
index=security user=admin OR user=root

# NOT
index=security NOT action=success
index=security user!=admin
index=security NOT (user=svc_* OR user=admin)

# Combined
index=security (EventCode=4624 OR EventCode=4625) 
  AND NOT user=SYSTEM 
  AND src_ip=10.*

Operator Precedence

# Precedence (highest to lowest): NOT, OR, AND
# Use parentheses to be explicit!

# Ambiguous:
index=security user=admin OR user=root AND action=failed
# Interpreted as: user=admin OR (user=root AND action=failed)

# Clear:
index=security (user=admin OR user=root) AND action=failed

Field Searching

Basic Field Searches

# Exact match
user=admin
EventCode=4624
src_ip=192.168.1.50

# Not equal
user!=admin
action!=success

# Field exists
user=*
# Field does not exist
NOT user=*

# Case sensitivity
# By default, field VALUES are case-sensitive
user=Admin       # Won't match "admin"
user=admin       # Exact match

# Case-insensitive search
| where lower(user)="admin"
# Or use CASE() function for case-sensitive
| where CASE(user)="Admin"

Wildcards

# Wildcard: * matches any characters
user=admin*           # Starts with admin
user=*admin           # Ends with admin  
user=*admin*          # Contains admin
src_ip=192.168.1.*    # IP range
src_ip=192.168.*.*    # Broader range

# Single character wildcard: NOT supported in basic search
# Use regex for single character matching

# Examples
index=security user=svc_*           # Service accounts
index=security host=web*            # Web servers
index=security src_ip=10.0.0.*      # Internal subnet

IN Operator

# Match multiple values
EventCode IN (4624, 4625, 4648)
user IN ("admin", "root", "administrator")
src_ip IN ("10.0.0.1", "10.0.0.2", "10.0.0.3")

# NOT IN
EventCode NOT IN (4624, 4634)
user NOT IN ("SYSTEM", "LOCAL SERVICE")

Comparison Operators

# Used with 'where' command (after pipe)
| where count > 10
| where bytes >= 1000000
| where duration < 60
| where user != "admin"

# String comparisons
| where like(user, "admin%")     # SQL-like wildcard
| where match(user, "^admin")    # Regex match

# Operators reference:
# =, ==, !=  - Equality
# >, >=      - Greater than
# <, <=      - Less than
# AND, OR    - Logical
# NOT, !     - Negation

Keyword and Raw Search

# Keyword search (searches _raw field)
index=security "failed login"
index=security error
index=security "password"

# Phrase search (exact phrase)
index=security "authentication failed"

# Keyword search is CASE-INSENSITIVE
index=security "ERROR"      # Matches "error", "Error", "ERROR"

# Combine keywords and fields
index=security "failed password" user=admin

# Escape special characters
index=security "user\=admin"     # Literal = sign
index=security "C:\\Windows"     # Backslash path

Subsearches

# Subsearch: Use results of one search in another
# Enclosed in square brackets []

# Find all events from users who failed login
index=security action=success 
  [search index=security action=failed 
   | stats count by user 
   | where count > 5 
   | fields user]

# Find events from IPs flagged in threat intel
index=firewall action=allowed
  [search index=threat_intel threat_score>80
   | fields src_ip 
   | rename src_ip as dest_ip]

# Subsearch returns field values to outer search
# Limit: 10,000 results, 60 second timeout by default
‚ö†Ô∏è Warning

Subsearches have limits. By default, subsearches return max 10,000 results and timeout after 60 seconds. For large datasets, use join or lookup instead. Also, subsearches run first and their results are substituted into the main search, which can create very large searches.

Search Modes

Splunk offers three search modes that affect speed and results:

Mode Behavior Use Case
Fast Field discovery disabled, only returns fields needed by search Dashboards, reports, known searches
Smart Auto-selects based on search type Default mode, general use
Verbose All field discovery enabled, shows all events and fields Investigation, exploring data

Common Security Search Patterns

# Failed logins in last 24 hours
index=security sourcetype=WinEventLog:Security EventCode=4625 earliest=-24h
| stats count by user, src_ip
| sort -count

# Successful logins from external IPs
index=security EventCode=4624 LogonType=10 NOT src_ip=10.* NOT src_ip=192.168.*
| table _time, user, src_ip, dest

# Process execution with command line
index=sysmon EventCode=1 earliest=-1h
| table _time, Computer, User, Image, CommandLine
| sort -_time

# Firewall denies by source
index=firewall action=deny earliest=-24h
| stats count by src_ip, dest_port
| sort -count
| head 20

# User activity timeline
index=* user=jsmith earliest=-7d
| table _time, index, sourcetype, action, src_ip, dest
| sort _time
üíº Interview Question

"How do you optimize a slow Splunk search?"

1. Specify index and sourcetype (most impactful)
2. Narrow time range
3. Move filters left (filter early, transform late)
4. Use indexed fields in search terms
5. Avoid wildcards at the start of terms
6. Use stats instead of transaction when possible
7. Consider summary indexing or data models for repeated searches

Search Best Practices

SEARCH OPTIMIZATION RULES:

1. ALWAYS specify index
   Bad:  sourcetype=WinEventLog:Security
   Good: index=security sourcetype=WinEventLog:Security

2. Specify time range
   Bad:  index=security user=admin
   Good: index=security user=admin earliest=-24h

3. Filter early, transform late
   Bad:  index=security | stats count by user | where user="admin"
   Good: index=security user=admin | stats count

4. Use specific terms before wildcards
   Bad:  index=security *admin*
   Good: index=security user=*admin*

5. Avoid leading wildcards
   Bad:  index=security user=*smith
   Good: index=security user=jsmith OR user=asmith

6. Use indexed fields when possible
   Indexed: index, source, sourcetype, host
   Also indexed: Any field extracted at index-time
üß† Memory Aid

SPL = Unix Pipes for Security Data

Think of SPL like a water treatment plant:
‚Ä¢ index= is the water source (where data comes from)
‚Ä¢ | (pipe) sends water to the next treatment stage
‚Ä¢ search/where = filters removing unwanted particles
‚Ä¢ stats/table = final packaging for delivery

Remember the Optimization Order:
"I-T-F-T" = Index ‚Üí Time ‚Üí Filter ‚Üí Transform
(Like making coffee: pick beans, set timer, filter grounds, pour into cup)

Chapter Summary

  • SPL searches flow through a pipeline: search terms ‚Üí commands ‚Üí commands
  • Always specify index and time range for efficient searches
  • Time modifiers: relative (-24h, -7d@d) or absolute (epoch, date strings)
  • Boolean operators: AND (implicit), OR, NOT with proper precedence
  • Field searches: exact match, wildcards (*), IN operator
  • Subsearches run first and substitute results into main search
  • Search modes: Fast (dashboards), Smart (default), Verbose (investigation)
  • Optimization: index first, time second, filter early, transform late
