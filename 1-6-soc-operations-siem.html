SOC Operations & SIEM

How Security Operations Centers use SIEM for alert triage, investigation, and incident response

ğŸ“˜ Chapter Overview

The SIEM is the central nervous system of a Security Operations Center. This chapter covers how SOC teams are structured, alert triage workflows, investigation procedures, escalation processes, and the key metrics that measure SOC effectiveness.

Tiered SOC Model

Most SOCs organize analysts into tiers based on skill level and responsibility:

Tier 1: Alert Triage (SOC Analyst)

First responders who handle initial alert review:

  • Monitor alert queues and dashboards
  • Perform initial triage using runbooks
  • Classify alerts: true positive, false positive, benign true positive
  • Escalate confirmed incidents to Tier 2
  • Document findings in case management system

Typical metrics: Alerts triaged per hour, escalation accuracy, queue age

Tier 2: Investigation (Security Analyst)

Experienced analysts who investigate escalated incidents:

  • Conduct deep-dive investigation
  • Correlate events across multiple sources
  • Determine scope and impact
  • Recommend or execute containment actions
  • Escalate complex incidents to Tier 3

Typical metrics: Mean time to investigate, containment success, false escalation rate

Tier 3: Advanced Response (Senior Analyst / IR)

Senior analysts and incident responders for complex threats:

  • Lead major incident response
  • Perform forensic analysis
  • Conduct threat hunting
  • Develop new detection rules
  • Mentor junior analysts

Typical metrics: Incidents resolved, detection rules created, hunt findings

ğŸ’¡ Modern SOC Evolution

Many organizations are moving away from strict tiering toward "T-shaped" analysts who specialize in one area but can handle diverse alerts. Automation (SOAR) handles routine Tier 1 tasks, allowing analysts to focus on investigation and hunting. Some SOCs now use "pods" of mixed-skill analysts rather than tiers.

Alert Triage Workflows

Effective triage quickly distinguishes threats from noise:

Triage Process

1. RECEIVE ALERT
   â””â”€â”€ Alert appears in queue with severity, description, and entities

2. INITIAL ASSESSMENT (30 seconds - 2 minutes)
   â”œâ”€â”€ Read alert title and description
   â”œâ”€â”€ Check severity and priority
   â”œâ”€â”€ Identify key entities (user, host, IP)
   â””â”€â”€ Check if alert is in known FP list

3. CONTEXT GATHERING (2-5 minutes)
   â”œâ”€â”€ Review related events in SIEM
   â”œâ”€â”€ Check entity history (user normal behavior?)
   â”œâ”€â”€ Query threat intelligence
   â”œâ”€â”€ Check asset criticality
   â””â”€â”€ Review similar past alerts

4. DETERMINATION
   â”œâ”€â”€ TRUE POSITIVE â†’ Document and escalate
   â”œâ”€â”€ FALSE POSITIVE â†’ Document reason, close, consider tuning
   â””â”€â”€ BENIGN TRUE POSITIVE â†’ Expected activity, document, close

5. DOCUMENTATION
   â””â”€â”€ Record findings, actions, and disposition in case notes

Triage Questions to Ask

  • Is this user/system expected to perform this activity?
  • What is the asset criticality of affected systems?
  • Is there threat intelligence on the indicators involved?
  • Are there related alerts or events suggesting a pattern?
  • What is the business context (maintenance window, known change)?

Example Triage Runbook

RUNBOOK: Brute Force Authentication Alert

ALERT: Multiple failed authentication attempts detected

TRIAGE STEPS:
1. Identify target account
   - Is this a service account, admin, or standard user?
   - Is the account currently locked out?

2. Check source information
   - Is source IP internal or external?
   - GeoIP lookup - is location expected?
   - Threat intel lookup on source IP

3. Review authentication pattern
   - How many failures? Over what time period?
   - Was there a subsequent success?
   - Multiple accounts targeted from same source?

4. Verify with user (if needed)
   - Did user forget password?
   - Did user recently travel?

DETERMINATION CRITERIA:
- TRUE POSITIVE: External IP, many failures, subsequent success, no user travel
- FALSE POSITIVE: User confirmed bad password, internal testing
- ESCALATE: Successful login from suspicious source, privileged account

RESPONSE ACTIONS:
- TP: Reset password, block source IP, escalate to Tier 2
- FP: Document, close, consider tuning if repeated

Investigation Procedures

When an alert is confirmed as suspicious, Tier 2 analysts conduct systematic investigation:

Investigation Framework

INVESTIGATION PHASES:

1. SCOPE DETERMINATION

   â”œâ”€â”€ What users/accounts are involved?
   â”œâ”€â”€ What data may be impacted?
   â””â”€â”€ What is the time range of activity?

2. TIMELINE CONSTRUCTION

   â”œâ”€â”€ What was the initial access vector?

   â””â”€â”€ Were systems modified?

Metrics and KPIs

Measure SOC performance with meaningful metrics:

Detection Metrics

Metric Definition Target
MTTD Mean Time to Detect - Time from attack to alert < 1 hour (varies by attack type)
Detection Coverage % of ATT&CK techniques with detections > 60% of relevant techniques
True Positive Rate % of alerts that are real threats > 70%

Response Metrics

Metric Definition Target
MTTA Mean Time to Acknowledge - Time from alert to first analyst action < 15 minutes (critical), < 1 hour (high)
MTTI Mean Time to Investigate - Time to complete investigation < 4 hours (typical)
MTTR Mean Time to Respond/Remediate - Time to containment < 24 hours (varies by severity)
MTTC Mean Time to Contain - Time to stop active threat < 1 hour (critical)

Operational Metrics

Metric Definition Target
Alert Volume Total alerts per day/week Manageable by team (analyst capacity)
Queue Age Oldest unacknowledged alert < 1 hour (critical), < 4 hours (all)
Alerts per Analyst Average alerts triaged per analyst per shift 20-40 (quality over quantity)
Escalation Accuracy % of escalations that were appropriate > 90% 
# Splunk - Calculate MTTA (Mean Time to Acknowledge)
index=notable_events status!="new"
| eval acknowledge_time = strptime(status_changed, "%Y-%m-%d %H:%M:%S")
| eval alert_time = _time
| eval time_to_acknowledge = (acknowledge_time - alert_time) / 60
| stats avg(time_to_acknowledge) as MTTA_minutes by urgency

# Alert volume trend
index=notable_events
| timechart span=1d count by urgency
// KQL - Calculate MTTA
SecurityIncident
| where Status != "New"
| extend TimeToAcknowledge = datetime_diff('minute', FirstModifiedTime, CreatedTime)
| summarize MTTA_Minutes = avg(TimeToAcknowledge) by Severity

// Alert volume trend
SecurityAlert
| summarize AlertCount = count() by bin(TimeGenerated, 1d), AlertSeverity
| render timechart
ğŸ’¼ Interview Question

"What SOC metrics do you track, and why are they important?"

Key metrics fall into three categories: (1) Detection - MTTD measures how quickly we detect threats, TP rate shows detection quality; (2) Response - MTTA, MTTI, MTTR measure how quickly we respond and resolve; (3) Operational - Queue age and alerts per analyst ensure sustainable workload. These metrics help identify bottlenecks, justify staffing, and demonstrate security program effectiveness to leadership.

Chapter Summary

  • SOCs typically use a tiered model: Tier 1 (triage), Tier 2 (investigation), Tier 3 (advanced response)
  • Alert triage follows a systematic process: receive, assess, gather context, determine, document
  • Investigations should determine scope, build timeline, assess impact, and plan containment
  • Clear escalation criteria and communication templates ensure appropriate response
  • Shift handoffs require structured documentation of open incidents and notable events
  • Key metrics include MTTD, MTTA, MTTI, MTTR, true positive rate, and queue age
