THREAT: Password Spray Attack

RESEARCH:
â€¢ Attack pattern: Many failed logins across multiple users from single IP
â€¢ Data source: Sign-in logs (Azure AD) or Security Events (Windows)
â€¢ Key indicators: ResultType != 0, many unique users, single source IP
â€¢ Threshold: 5+ targeted users within 10 minutes is suspicious

DETECTION LOGIC (KQL):

// Password Spray Detection
let TimeWindow = 10m;
let UserThreshold = 5;
let FailureThreshold = 10;
// Exclusions
let ExcludedIPs = _GetWatchlist('approved_ips') | project IPAddress;
let ServiceAccounts = _GetWatchlist('service_accounts') | project UserPrincipalName;
// Detection query
SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType != 0                          // Failed logins
| where ResultType == "50126"                    // Invalid password
| where IPAddress !in (ExcludedIPs)              // Exclude known IPs
| where UserPrincipalName !in (ServiceAccounts)  // Exclude service accounts
| summarize 
    TargetedUsers = dcount(UserPrincipalName),
    UserList = make_set(UserPrincipalName, 10),
    TotalAttempts = count(),
    FirstAttempt = min(TimeGenerated),
    LastAttempt = max(TimeGenerated)
    by IPAddress, bin(TimeGenerated, TimeWindow)
| where TargetedUsers >= UserThreshold
| where TotalAttempts >= FailureThreshold
| extend Description = strcat("Password spray detected from ", IPAddress, 
         " targeting ", TargetedUsers, " users with ", TotalAttempts, " attempts")
| project TimeGenerated, IPAddress, TargetedUsers, UserList, TotalAttempts, Description

ğŸ’¼ Interview Question

"Walk me through how you would develop a new detection rule."

1. Identify the Threat: Start with threat intelligence, incident findings, or coverage gaps. Understand what attack technique I'm trying to detect.
2. Research: Study the attack, identify required data sources, review existing detections (SIGMA, vendor content).
3. Develop: Write the query with clear thresholds, include exclusions for known false positives, add entity mappings for investigation.
4. Test: Validate with attack simulation (atomic tests), run against historical data, check performance.
5. Document: Create detection documentation with MITRE mapping, response procedures, and false positive guidance.
6. Deploy & Monitor: Enable in production, track effectiveness, tune based on feedback.

Chapter Summary

  • Detection development follows a lifecycle: identify, research, develop, test, deploy, monitor
  • Three detection approaches: signature, behavioral, anomaly
  • Detection structure: data source, filter, threshold, exclusions, enrichment
  • Common patterns: threshold, first-time, sequence, time-based, peer outlier
  • Documentation is critical for maintenance and SOC usage
  • Always include exclusion lists for tuning
